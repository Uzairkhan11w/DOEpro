# Deployment, publishing, DOI and citation

Reference notes for maintainers of **DOEpro**. Two goals, handled separately:

- **Reach** — put the app in front of as many students as possible, free, with no
  maintenance burden.
- **Credit** — make the work citable: CRAN, GitHub, a DOI, and a peer-reviewed software
  paper.

under, and `10.5281/zenodo.XXXXXXX` for the Zenodo concept DOI once it is minted.

## 1. Hosting

### Primary: shinylive on GitHub Pages — free, serverless

`shinylive` compiles the app to WebAssembly, so R runs **inside the visitor's browser**.
There is no server: no bill, no session limit, no concurrency ceiling, nothing to keep
alive. It scales to any number of simultaneous users at zero cost.

This is why the app depends only on `shiny`, `DT` and `ggplot2` — all three have
WebAssembly builds. Heavy or compiled-only packages are the usual cause of shinylive
failures, and none are used here.

```r
install.packages("shinylive")
# with app.R in a folder, e.g. app/
shinylive::export("app", "docs")
# commit docs/ and enable GitHub Pages: Settings -> Pages -> Deploy from branch -> main -> /docs
```

The app is then served at `https://uzairkhan11w.github.io/DOEpro/`.

Properties worth noting:

- First load pulls roughly 30–60 MB of WebAssembly; it is cached afterwards.
- There is no server-side R process, so **server-side PDF export is unavailable** in this
  mode. The HTML report download works, and the browser's own *Print → Save as PDF*
  produces the PDF; the page footer carries the credit line either way.
- Uploads and downloads stay on the visitor's machine — a genuine privacy advantage for
  unpublished trial data.

### Alternative: a hosted R server (enables server-side PDF export)

| Option | Free tier | Notes |
|---|---|---|
| Posit Connect Cloud | free for public apps, deploys from GitHub | easiest server-backed route; add a `manifest.json` |
| Hugging Face Spaces | free CPU Space, Docker | install `wkhtmltopdf` in the Dockerfile and PDF export works |
| shinyapps.io | 5 apps, 25 active hours/month | fine for a single class; limited for a wide audience |
| Institutional Shiny Server | free if hosted by the institution | Shiny Server Open Source on a departmental subdomain |

### Minimal-cost scaling

Google Cloud Run or fly.io (container, scales to zero, a few dollars a month at teaching
scale), or a small DigitalOcean droplet running ShinyProxy.

### Recommended arrangement

Publish **shinylive on GitHub Pages** as the primary front door, and run **one Hugging Face
Space** as a mirror for users who want server-side PDF export. Link both from the README.

## 2. R package structure

CRAN requires a package, not a bare `app.R`:

```
DOEpro/
├── DESCRIPTION
├── NAMESPACE            # generated by roxygen2
├── LICENSE / LICENSE.md # GPL-3
├── NEWS.md
├── R/
│   ├── helpers.R        # fmt, star, cld_lsd, read_pasted, df_html
│   ├── anova.R          # analyze, tidy_aov, new_effect, run_all
│   ├── assumptions.R    # levene_test, boxcox_profile, check_assumptions, suggest_transform
│   ├── posthoc.R        # posthoc, PH_METHODS
│   ├── tables.R         # sketch_main_html, sketch_twoway_html, means_section_html, combined_anova_html
│   ├── plots.R
│   ├── report.R         # build_report, save_pdf
│   └── run_app.R        # run_app()
├── inst/
│   ├── CITATION
│   └── app/app.R        # the Shiny app, calling the package's exported functions
├── man/                 # roxygen2 output
├── tests/testthat/
└── vignettes/DOEpro.Rmd
```

```r
run_app <- function(...) {
  shiny::runApp(system.file("app", package = "DOEpro"), ...)
}
```

Package-name availability is checked with:

```r
install.packages("available")
available::available("DOEpro")
```

The name is chosen to stand on its own and not to imply a fork of, or affiliation with, any
existing named package; the documentation presents DOEpro as an independent tool.

### CRAN checklist

- `R CMD check --as-cran` with zero errors, warnings and notes.
- No writing outside `tempdir()`. The app writes only to `tempfile()`.
- Examples run in a few seconds; the app launcher is wrapped in `\donttest{run_app()}`.
- `pagedown` stays in **Suggests**, guarded by `requireNamespace()`.
- `NEWS.md` and `cran-comments.md` present.
- Checked on r-devel, Windows, macOS and Linux before submission
  (`usethis::use_github_action("check-standard")` or `rhub::rhub_check()`).

### Statistical regression tests

The verified numerical properties make good `testthat` cases and are the ones a reviewer
will look for:

```r
test_that("Duncan R2 equals Fisher's LSD", { ... })        # tolerance 1e-6
test_that("SNK R_k equals Tukey HSD", { ... })
test_that("Box-Cox profile matches the explicit-Jacobian likelihood", { ... })
test_that("split-plot df are r-1, a-1, (r-1)(a-1), b-1, (a-1)(b-1), a(r-1)(b-1)", { ... })
test_that("all four split-plot SE(d) match the closed-form expressions", { ... })
test_that("every transformation round-trips through its inverse", { ... })
```

## 4. DOI (Zenodo)

Zenodo mints DOIs for GitHub releases, free and permanently.

1. Sign in to Zenodo with GitHub and enable the `DOEpro` repository.
2. Create a GitHub **Release** (tag `v2.0.0`).
3. Zenodo archives the release and issues two DOIs: a **version DOI** for `v2.0.0`, and a
   **concept DOI** that always resolves to the newest version. Publications cite the
   concept DOI.
4. The concept DOI replaces the `10.5281/zenodo.XXXXXXX` placeholder in `CITATION.cff`,
   `inst/CITATION` and the README badge.

Zenodo reads author and title metadata from `CITATION.cff`.

### Software paper

- **JOSS** (Journal of Open Source Software): free, open peer review, its own DOI and a
  short citable paper. Requires an open licence, substantial scholarly effort,
  documentation, tests and a `paper.md`. Submitted after the CRAN release and Zenodo DOI
  exist.
- Alternatives: *SoftwareX* (Elsevier, APC), or an applications note in an Indian
  agricultural-statistics journal.
